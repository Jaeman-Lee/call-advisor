# CPU-only 로컬 음성 전사 평가

## 목적

고성능 GPU나 외부 음성 API 없이 스마트폰 및 일반 IP폰급 ARM CPU에서 통화 음성을
실용적으로 전사할 수 있는 최소·권장 자원을 실측합니다. 원음은 기기 밖으로 보내지
않고, 로컬 전사 후 생성된 텍스트만 선택적으로 Codex 분석에 사용합니다.

## 보안 경계

```text
원본 통화 음성 ──► FFmpeg ──► Silero VAD ──► whisper.cpp
     기기 내부에서만 처리                         │
                                                  ▼
                                             전사 텍스트
                                                  │
                                      선택적으로 Codex에 전달
```

완전 오프라인이 필요하면 Codex 단계를 끄고 전사문만 저장하거나, 로컬 요약 모델로
교체해야 합니다. 현재 구조는 음성 자체의 외부 전송을 제거하지만 전사 텍스트는 Codex로
전달하므로 통화 내용에 대한 클라우드 노출이 완전히 사라지는 것은 아닙니다.

완전 로컬 모드는 개인 설정에서 다음과 같이 활성화합니다.

```sh
CODEX_ENABLED=false
```

이 경우 원음과 전사 텍스트 모두 기기 밖으로 나가지 않으며 로컬 전사문과 성능
CSV만 생성됩니다.

## 측정 지표

- 전체 오디오 길이와 VAD 발화 구간 길이
- 발화 밀도와 무음 절감 가능 비율
- Whisper wall time과 재생시간 대비 전사 속도
- 평균 사용 코어, CPU core-seconds, 최대 RSS
- 오디오 1분 및 발화 1분당 CPU 사용량
- 전사 문자 수와 발화 1분당 문자 수
- Codex 분석 및 전체 처리시간

## 리소스 프로파일

같은 음성을 스레드 1·2·4·6개와 VAD on/off로 반복해 CSV를 출력합니다.

```sh
./benchmark-local.sh '/storage/emulated/0/Recordings/Call/example.m4a' \
  > benchmark.csv
```

평가 기준:

- 최소 구성: 1.0x realtime 이상이면서 가장 적은 코어·메모리
- 균형 구성: 처리시간과 CPU core-seconds가 함께 낮은 지점
- 최대 처리 구성: 발열 전 짧은 구간에서 가장 높은 realtime speed

짧은 통화 한 건만으로 장시간 성능을 확정하면 안 됩니다. 1분, 5분, 15분 샘플을
각각 시험하고 배터리 온도와 Android 백그라운드 cpuset 제한을 함께 기록해야 합니다.

## 현재 기기 실측 결과

18.09초 한국어 통화 한 건을 기준으로 한 초기 결과입니다. 정확한 최소 사양을
확정하려면 더 긴 음성과 다양한 화자를 추가해야 합니다.

### 자동 CPU 스케줄링, base 모델

| 스레드 | VAD | wall time | 평균 코어 | 최대 RSS | 전사 속도 |
|---:|:---:|---:|---:|---:|---:|
| 1 | off | 11.39초 | 0.99 | 284.0MB | 1.59배속 |
| 2 | off | 10.28초 | 1.95 | 284.7MB | 1.76배속 |
| 4 | off | 6.29초 | 3.89 | 285.6MB | 2.88배속 |
| 6 | off | 5.32초 | 5.72 | 285.7MB | 3.40배속 |

발화 밀도가 약 85%인 이 샘플에서는 VAD on이 모든 스레드 구성에서 조금 느렸습니다.
그래서 운영 로직은 VAD로 밀도를 먼저 측정하되 기본 80% 이하일 때만 Whisper 입력
압축을 적용합니다.

| 모델 | 코어 | wall time | 최대 RSS | 전사 속도 | 품질 판단 |
|---|---:|---:|---:|---:|---|
| base | 1 | 66.89초 | 283.0MB | 0.27배속 | 기준 품질 |
| base | 2 | 37.61초 | 283.1MB | 0.48배속 | 기준 품질 |
| base | 4 | 27.36초 | 285.3MB | 0.66배속 | 기준 품질 |
| tiny | 2 | 22.49초 | 139.4MB | 0.80배속 | 한국어 오류 증가 |
| tiny | 4 | 21.04초 | 147.4MB | 0.86배속 | 한국어 오류 증가 |
| tiny-q5 | 4 | 17.39초 | 129.8MB | 1.04배속 | 업무용 품질 미달 |
| base-q5 | 4 | 36.07초 | 195.4MB | 0.50배속 | 품질 유지, 속도 감소 |

### 현재 결론

- 최소 메모리 실시간 후보: A520급 4코어 + tiny-q5 + 약 130MB RAM. 다만 이
  샘플에서 한국어 업무 전사 품질이 부족해 제품 기준으로 채택할 수 없습니다.
- 최소 품질 후보: base + 약 285MB RAM. A520 4코어만으로는 0.66배속이므로
  통화 후 배치 처리에는 가능하지만 실시간에는 부족합니다.
- 권장 균형점: base + 성능 코어 포함 2~4스레드 + 약 300MB RAM. 현재 스마트폰은
  1.76~2.88배속을 달성했습니다.
- base-q5는 RAM과 저장공간이 제한된 장치에는 의미가 있지만 이 CPU에서는
  dequantization 비용으로 더 느렸습니다.

일반 IP폰급 장치에서 품질과 실시간성을 모두 원하면 Cortex-A7xx급 성능 코어,
전용 NPU/DSP, 또는 더 효율적인 한국어 특화 모델 중 하나가 필요할 가능성이 큽니다.

동일 기기에서도 Android가 Termux를 백그라운드 효율 코어로 제한한 실측에서는
base/6스레드가 31.98초에 18.09초 음성을 처리해 0.57배속에 그쳤습니다. 포그라운드
실측 3.40배속과 큰 차이가 있으므로 제품 사양에는 CPU 모델뿐 아니라 OS 스케줄링,
foreground service 정책과 지속 발열 조건을 포함해야 합니다.
